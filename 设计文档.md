# SQL 优化工具设计文档

# 一、项目定位

本项目用于解决 BI / 报表平台自动生成 SQL 难以人工改写的问题。工具接收原始 BI SQL，识别是否属于已知低效 SQL 模板；如果命中规则，则自动改写成预先验证过的提速 SQL 模板；如果未命中规则或改写风险较高，则返回原 SQL，不强行优化。

第一版对外定位为 **BI 重点慢报表 SQL 自动改写加速器**。

第一版目标不是做通用 SQL 优化器，也不是替代数据库优化器，更不是自动生成“最优 SQL”的智能系统，而是做一个**规则驱动、可审计、可回退的 SQL 自动改写工具**。它只对已采集、已分析、已验证结果一致的 SQL 模板生效。

适用场景：

- BI 平台自动生成 SQL，业务侧无法直接改写。
- 慢 SQL 模式相对稳定，可以归纳成模板。
- 已经有人工验证过的提速 SQL 模板。
- 需要自动识别原 SQL，并替换成等价、高性能版本。

不适合场景：

- 任意未知 SQL 的自动优化。
- 自动发明提速 SQL。
- 结果口径无法校验的 SQL。
- 需要修改业务语义才能提升性能的 SQL。
- 没有已验证提速 SQL 模板可替换的场景。

## 1.1 对外表述

对客户表达时，不建议称为“SQL 自动优化器”或“自动生成最优 SQL 的工具”。这类说法容易让客户误解为任意 SQL 都能被自动优化，最终导致验收边界失控。

建议对外表述为：

- BI 重点慢报表 SQL 自动改写加速器。
- 规则驱动的 BI 慢 SQL 加速工具。
- 已知慢 SQL 模板的自动替换和验证工具。

推荐一句话介绍：

**针对 BI 平台自动生成、业务侧无法直接修改的重点慢 SQL，本工具通过规则识别和模板替换，把已验证过的提速 SQL 模板自动生成出来；未命中或存在风险的 SQL 保持原样，确保可验证、可审计、可回退。**

## 1.2 客户价值

客户真正关心的不是工具内部如何实现，而是以下结果：

- 哪些重点报表会变快。
- 覆盖多少条慢 SQL 或多少类慢 SQL 模板。
- 改写前后结果是否一致。
- 每条 SQL 性能提升多少。
- 未命中 SQL 是否保持原样。
- 出现问题时是否可以关闭规则并回退。
- 是否能形成可交付的性能对比报告和一致性校验报告。

因此，第一版交付不能只展示 CLI 能跑通，还要输出“客户可验收”的证据，包括命中清单、改写 SQL、结果一致性、性能对比和审计日志。

## 1.3 研发边界

研发实现必须遵守以下边界：

- 只支持已采集的 BI SQL 样本，不支持任意 SQL。
- 只支持确定性模板改写，不自动发明 SQL。
- 只对通过结果一致性校验的规则启用自动改写。
- 第一版只做 CLI + 改写引擎，不做 HTTP 服务，不做 PostgreSQL 协议代理。
- 不承诺 BI SQL 结构轻微变化后仍然自动适配，结构变化需要重新采样和评估规则。
- 参数抽取必须类型化，不能把正则捕获结果不经处理直接拼进 SQL。
- 指纹匹配从严格开始，宁可漏命中，也不能误命中。
- 性能目标按具体 SQL 样本和指定测试环境验收，不泛化承诺所有 SQL 都能提升。

# 二、语言选择

项目使用 **Go** 开发。

选择 Go 的原因：

- 编译型语言，性能足够高，适合后续做常驻服务或协议代理。
- 部署简单，可以编译成单个二进制文件。
- 并发模型简单，适合后续扩展成 PostgreSQL 协议代理。
- 标准库完善，处理 CLI、配置、日志、网络服务都比较直接。
- 开发效率高，第一版可以较快落地。

不选择 Rust 的原因：

- Rust 性能更强，但第一版开发成本更高。
- 本项目第一阶段重点是规则体系、参数抽取、结果校验和可观测性，不是极限性能。

不选择 Python 的原因：

- Python 适合快速验证，但后续做高并发代理和单二进制部署不如 Go。
- 如果后续要放在 BI 和数据库之间常驻运行，Go 更合适。

# 三、第一版目标

第一版做成 CLI 工具和可复用的改写引擎，不直接做 HTTP 服务，也不直接做 PostgreSQL 协议代理。

第一版成功的标准是：用某客户场景 13 条已采集 BI SQL 样本跑通“识别、改写、解释、校验、审计、回退”的完整闭环。

第一版交付价值是离线证明：哪些重点慢报表 SQL 可以通过规则化改写稳定提速，并沉淀下一阶段 HTTP 服务或 PostgreSQL 协议代理所需的规则资产、模板资产、校验证据和性能对比证据。第一版不直接接入 BI 线上流量。

核心能力：

1. 读取一条原始 SQL。
2. 对 SQL 做规范化处理。
3. 生成 SQL 指纹。
4. 根据规则库判断是否命中可改写模板。
5. 从原 SQL 中抽取参数。
6. 渲染已验证提速 SQL 模板。
7. 输出改写后的 SQL。
8. 未命中时返回原 SQL。
9. 可选连接数据库，校验原 SQL 和改写 SQL 的结果一致性。
10. 输出命中规则、改写原因、参数、风险提示和审计日志。
11. 可选输出 DDL 和物理设计建议，例如字段类型、建表方式、分布键、排序键、索引和统计信息建议。

## 3.1 第一版范围锁定

第一版范围锁定为某客户场景 13 条已采集 BI SQL 样本。除非重新评审设计文档和实现计划，否则不得在第一版中临时扩展到其它客户、其它报表或其它未知 SQL。

每条样本必须进入验收清单，至少包含以下字段：

| 字段 | 说明 |
|---|---|
| `sql_id` | 样本编号，例如 `SQL_001`。 |
| `report_name` | 对应报表或图表名称。 |
| `owner` | 业务确认人或需求来源。 |
| `raw_sql_file` | 原始 SQL 文件路径。 |
| `expected_rule` | 预期命中的规则名；未纳入改写则填 `none`。 |
| `expected_template` | 预期使用的模板路径；未纳入改写则填 `none`。 |
| `expected_params` | 预期抽取的参数清单。 |
| `expected_action` | `rewrite`、`original` 或 `dry_run_only`。 |
| `validation_required` | 是否必须做结果一致性校验。 |
| `acceptance_scope` | 是否纳入客户验收。 |
| `anonymized` | 样本 SQL 是否完成脱敏。 |

如果 13 条中存在暂时无法连接数据库做 `validate` 的 SQL，该 SQL 可以做 `fingerprint`、`explain` 和 `rewrite` 演示，但必须标记为 `dry_run_only`，不能标记为建议进入自动改写候选规则库。

## 3.2 样本 SQL 脱敏要求

所有进入 `samples/`、`testdata/`、规则文件、模板文件、验收报告和 Git 仓库的 SQL 样本都必须脱敏。不得提交真实客户名称、真实用户、真实账号、真实连接串、真实组织结构或其它可识别业务主体的信息。

必须脱敏的内容包括：

- 客户名称、项目名称、港口/公司/部门等可识别主体。
- `userid`、`user_id`、`account_id`、`tenant_id`、`org_id`、`company_id` 等身份字段的真实值。
- 用户名、姓名、手机号、邮箱、身份证号、工号、登录账号。
- 数据库地址、端口、用户名、密码、token、DSN。
- 业务合同号、订单号、客户号、船名、车牌、设备编号等敏感业务标识。

推荐占位方式：

| 原始类型 | 占位示例 |
|---|---|
| 用户 ID | `USER_001`、`USER_002` |
| 租户 ID | `TENANT_001` |
| 组织 ID | `ORG_001` |
| 公司 ID | `COMPANY_001` |
| 账号 | `ACCOUNT_001` |
| 客户名称 | `CUSTOMER_A` |
| 连接串 | `postgres://USER:PASSWORD@HOST:PORT/DBNAME` |

脱敏要求：

- 脱敏后 SQL 结构必须保持不变，不能因为脱敏改变规则匹配结果。
- 同一 SQL 内相同真实值必须替换为相同占位值，避免破坏等值条件。
- 不同真实值不能全部替换为同一个占位值，避免破坏 `IN` 列表和分组语义。
- 验收报告中只能记录脱敏后的参数摘要。
- 审计日志不得记录未脱敏的 SQL 参数。

第一版明确不做：

- 不做任意 SQL 的自动优化。
- 不做自动生成最佳执行计划。
- 不做在线生产流量代理。
- 不做规则管理 Web 平台。
- 不做未经结果一致性校验的自动改写候选。

第一版命令示例：

```bash
sqlopt rewrite --sql-file samples/raw/bi_001.sql
sqlopt fingerprint --sql-file samples/raw/bi_001.sql
sqlopt explain --sql-file samples/raw/bi_001.sql
sqlopt validate --sql-file samples/raw/bi_001.sql --dsn postgres://user:pass@host:5432/db
sqlopt advise --sql-file samples/raw/bi_001.sql --schema-file samples/schema/customer_schema.sql
```

# 四、整体架构

整体流程如下：

```text
原始 BI SQL
  -> SQL 规范化
  -> SQL 指纹生成
  -> 规则库匹配
  -> 参数抽取
  -> 已验证提速 SQL 模板渲染
  -> 输出改写 SQL / 原 SQL
  -> 可选生成 DDL / 物理设计建议
  -> 审计日志
  -> 可选结果一致性校验
```

模块划分：

| 模块 | 职责 |
|---|---|
| `cmd/sqlopt` | CLI 入口，负责解析命令行参数。 |
| `internal/normalize` | SQL 规范化，减少空格、大小写、常量、别名差异对识别的影响。 |
| `internal/fingerprint` | 基于规范化 SQL 生成稳定指纹。 |
| `internal/rules` | 加载规则文件，判断 SQL 是否命中某条规则。 |
| `internal/extract` | 从原 SQL 中抽取年份、机械类型、货物中类等参数。 |
| `internal/rewrite` | 根据模板和参数生成已验证提速 SQL。 |
| `internal/advisor` | 基于 SQL、DDL 和执行计划输出字段类型、建表、索引、统计信息等建议。 |
| `internal/validate` | 对原 SQL 和改写 SQL 做结果一致性校验。 |
| `internal/audit` | 记录改写命中、失败、回退、耗时等信息。 |
| `internal/config` | 加载全局配置。 |

第一版建议优先使用成熟 PostgreSQL SQL parser 或 AST 能力识别 SQL 结构。正则只能用于 AST 或结构特征确认后的局部参数抽取，不能承担完整 SQL 理解、结构判断或等价性判断。

# 六、核心概念

## 6.1 原始 SQL

原始 SQL 指 BI / 报表平台发送给数据库的 SQL。第一版先从文件读取，后续协议代理阶段再从网络连接中读取。

## 6.2 SQL 规范化

规范化的目标是让同一类 SQL 在不同常量、空格、大小写、别名变化下仍能生成稳定指纹。

第一版规范化包括：

- 去除首尾空白。
- 合并连续空白字符。
- 统一关键字大小写。
- 将字符串常量替换成占位符。
- 将数字常量替换成占位符。
- 保留表名、字段名、函数名和结构特征。

规范化不能把不同业务语义的 SQL 合并成同一类。对于 `join` 结构、聚合函数、分组字段、窗口函数、`distinct`、`limit`、`order by` 等可能影响结果口径的结构，第一版应保持严格识别。

## 6.3 SQL 指纹

SQL 指纹用于识别“这条 SQL 属于哪一类模板”。第一版可以使用规范化 SQL 的 SHA256 哈希，也可以叠加结构特征，例如目标表、过滤字段、聚合函数、分组字段。

指纹需要稳定，但不能过度宽松。宁可少命中，也不能误命中。

## 6.4 规则

规则描述“什么样的 SQL 可以被改写，以及改写成什么”。

规则内容包括：

- 规则名称。
- 适用客户或项目。
- 原 SQL 指纹。
- 必须出现的表或视图。
- 必须出现的过滤字段。
- 参数抽取方式。
- 已验证提速 SQL 模板路径。
- 是否启用自动改写。
- 风险等级。
- 回退策略。
- 结果一致性校验状态。
- 是否允许进入自动改写候选规则库。
- 规则版本号。
- 模板 checksum。
- 结构约束。
- 参数 schema。

规则示例：

```yaml
rules:
  - name: customer_report_summary_v1
    id: customer_case.report_summary.v1
    version: 1
    enabled: true
    project: customer_case
    match:
      fingerprints:
        - "sha256:example"
      required_relations:
        - "customer_report_fact"
      required_filters:
        - "年份"
      required_structure:
        statement_type: select
        forbid_distinct: true
        forbid_limit: true
        forbid_window_functions: true
        required_aggregates:
          - sum
          - max
    extract:
      年份:
        type: string
        required: true
        max_length: 4
        allowed_pattern: "^20[0-9]{2}$"
        pattern: "\"年份\"\\s*=\\s*'([^']+)'"
      机械类型:
        type: string_list
        required: false
        max_items: 50
        item_max_length: 64
        pattern: "\"机械类型\"\\s+IN\\s*\\(([^)]+)\\)"
    rewrite:
      template: templates/customer_case/report_summary_v1.sql
      template_checksum: "sha256:example-template"
    validation:
      required: true
      status: passed
      numeric_tolerance: 0.000001
      validated_at: "2026-07-03"
      validated_by: "manual"
    fallback:
      on_no_match: original
      on_extract_error: original
      on_render_error: original
```

## 6.5 规则匹配硬约束

规则匹配必须同时满足指纹和结构约束。只匹配表名或过滤字段不足以触发改写。

第一版匹配要求：

- SQL 类型必须是只读 `SELECT`。
- 规范化 SQL 指纹必须命中规则定义的指纹之一。
- 必须出现规则定义的关系名。
- 必须出现规则定义的关键过滤字段。
- `select list`、`join`、`where`、`group by`、聚合函数、`distinct`、`limit`、`order by`、窗口函数等结构必须满足规则约束。
- 禁止 DDL、DML、临时表创建、函数调用副作用等非只读语句进入改写路径。

冲突处理：

- 0 条规则命中：返回原 SQL，`action=original`，`reason=no_match`。
- 1 条规则命中但 `validation.status != passed`：只允许 `dry_run`，不能生成建议进入候选规则库的结果。
- 多条规则命中：返回原 SQL，`action=original`，`reason=ambiguous_match`。
- 指纹命中但结构约束失败：返回原 SQL，`reason=structure_mismatch`。
- 同一指纹对应多个启用规则：启动时规则加载失败，要求人工消除冲突。

## 6.6 参数 schema

每个参数都必须有 schema。参数 schema 至少包含：

- 参数名。
- 是否必填。
- 类型：`string`、`integer`、`decimal`、`date`、`timestamp`、`string_list`、`integer_list`。
- 允许格式或允许值。
- 最大长度。
- 列表最大元素数。
- 数值范围。
- 日期格式。
- 重复参数冲突规则。

参数抽取规则：

- 必填参数缺失：回退原 SQL。
- 同一参数出现多次且值不一致：回退原 SQL。
- 字符串超过最大长度：回退原 SQL。
- 列表超过最大元素数：回退原 SQL。
- `IN` 列表必须逐项解析、逐项校验、逐项转义，不能把原始 SQL 片段整段写入模板。
- 参数值只能来自原 SQL，不能由工具猜测或补默认业务值。

# 七、改写策略

第一版只支持**确定性模板改写**。

也就是说，工具不会自己发明 SQL，而是基于已经准备好的已验证提速 SQL 模板进行参数替换。

改写原则：

1. 只改写命中规则的 SQL。
2. 只改写结果口径已经验证一致的 SQL。
3. 只替换查询结构，不改变业务筛选条件。
4. 参数必须来自原 SQL，不能凭空生成。
5. 如果参数抽取失败，直接回退原 SQL。
6. 如果模板渲染失败，直接回退原 SQL。
7. 默认不对未知 SQL 做任何改写。
8. 改写后 SQL 的列名、列顺序、类型兼容性必须与原 SQL 满足验收要求。
9. 规则必须能解释命中原因，不能只给出“已改写”的黑盒结论。

## 7.1 数据库友好 SQL 原则

用户要求改写后的 SQL 必须是“数据库舒服”的 SQL。这里的“数据库舒服”不是语法更短，也不是看起来更整齐，而是让数据库优化器更容易选择稳定计划，让执行器更少做无效工作。

第一版的已验证提速 SQL 模板必须尽量满足以下原则：

- 过滤条件前置，让年份、货类、机械类型等高选择性条件尽早生效。
- 避免在过滤列、关联列、分组列上做不必要函数包裹，减少无法下推或无法利用统计信息的表达式。
- 避免隐式类型转换，模板参数类型必须和目标字段类型一致。
- 避免重复扫描同一大宽表，能合并扫描就合并，能复用中间结果就复用。
- 避免把宽 `text` 和高精度 `numeric` 字段无意义地带入中间结果。
- 聚合前尽量减少输入行数和输入列数。
- 关联顺序和关联键应尽量贴合数据分布，减少大规模 `Redistribute Motion`。
- 分组键和过滤键应尽量贴合底层表的分布键、排序键、分区键或 MARS block pruning 条件。
- 对报表只需要最终聚合结果的场景，避免输出明细宽行再在上层聚合。
- 避免 `select *`，模板必须显式列出需要的字段。
- 避免不必要的 `distinct`、全局排序、宽表 `order by` 和不可控的笛卡尔放大。
- 对需要排序的结果，排序字段必须和业务输出要求一致，不能为了性能删除业务必需排序。
- 对金额、吨位、成本等指标，字段类型和精度必须满足业务口径，同时避免过度高精度导致聚合成本过高。

模板评审时必须回答以下问题：

| 问题 | 要求 |
|---|---|
| 是否减少扫描行数？ | 说明过滤条件如何提前生效。 |
| 是否减少扫描列数？ | 说明模板为什么只读取这些列。 |
| 是否减少 Motion？ | 说明关联键、分组键和分布键的关系。 |
| 是否减少类型转换？ | 说明参数类型和字段类型是否一致。 |
| 是否减少宽行中间结果？ | 说明中间结果列宽是否比原 SQL 更小。 |
| 是否稳定利用统计信息？ | 说明关键过滤列、关联列、分组列是否需要 `ANALYZE`。 |

如果某条模板只是“写法不同”，但不能解释为什么更适合数据库执行，则不能进入 `candidate`。

数据库友好检查项必须结构化记录，不能只写自由文本：

| 检查项 | 允许值 | 证据要求 |
|---|---|---|
| `reduces_scan_rows` | `yes` / `no` / `not_applicable` | 说明过滤条件、分区裁剪或 block pruning 证据。 |
| `reduces_scan_columns` | `yes` / `no` / `not_applicable` | 说明读取列清单和原 SQL 对比。 |
| `reduces_motion` | `yes` / `no` / `not_applicable` | 说明 `Redistribute Motion` / `Broadcast Motion` 变化。 |
| `reduces_casts` | `yes` / `no` / `not_applicable` | 说明字段类型和参数类型是否一致。 |
| `reduces_wide_intermediate_rows` | `yes` / `no` / `not_applicable` | 说明中间结果列宽或字段数变化。 |
| `uses_stats_stably` | `yes` / `no` / `not_applicable` | 说明统计信息、估算行数和实际行数情况。 |

进入 `candidate` 的 SQL 必须有 `plan_evidence`，且上述检查项中至少一项为 `yes`，不能全部为 `no` 或 `not_applicable`。

## 7.2 DDL 与物理设计建议

有些慢 SQL 仅靠 SQL 文本改写无法达到理想效果。工具可以额外输出 DDL 和物理设计建议，但第一版只做建议，不自动执行 DDL，也不自动修改数据库对象。

DDL / 物理设计建议的范围：

- 字段类型建议：例如高精度 `numeric` 是否可降精度、字符串维度是否可改为更窄类型、过滤列和关联列类型是否需要统一。
- 建表建议：例如是否需要报表专用明细表、预聚合表、物化表或保持同名对象替换。
- 分布键建议：例如关联键、分组键、过滤键是否更适合作为分布键，是否存在数据倾斜风险。
- 排序键 / 分区键建议：例如年份、月份、货类等过滤字段是否适合作为排序或分区前缀。
- 索引建议：例如下拉筛选、点查或高选择性过滤是否适合 B-tree 索引；大范围聚合扫描不默认推荐普通索引。
- 统计信息建议：例如关键列是否需要提高 statistics target，是否需要重新 `ANALYZE`。
- 存储布局建议：例如宽表、列存、压缩、MARS block pruning 或 heap/MARS 对比是否需要专项验证。

每条建议必须包含以下字段：

| 字段 | 说明 |
|---|---|
| `recommendation_id` | 建议编号。 |
| `category` | `field_type`、`table_design`、`distribution_key`、`sort_key`、`partition`、`index`、`statistics`、`storage_layout`。 |
| `target_object` | 影响的表、视图、列或索引。 |
| `problem` | 当前问题，例如隐式转换、宽行、Motion、统计信息失真。 |
| `recommendation` | 建议动作。 |
| `expected_benefit` | 预期收益。 |
| `risk` | 精度、口径、写入成本、维护成本或回滚风险。 |
| `validation_method` | 如何验证结果一致性和性能收益。 |
| `rollback_plan` | 如何回滚。 |
| `execution_policy` | 第一版固定为 `manual_only`。 |

DDL 建议示例：

```yaml
recommendations:
  - recommendation_id: field_type_cost_amount_v1
    category: field_type
    target_object: customer_report_fact.cost_amount
    problem: "聚合字段精度过高，执行计划中 numeric 聚合 CPU 成本较高"
    recommendation: "在业务精度允许时，将报表层字段从 numeric(38,15) 调整为 numeric(20,6)"
    expected_benefit: "降低聚合 CPU 成本和中间状态宽度"
    risk: "可能影响金额展示精度，必须由业务确认误差阈值"
    validation_method: "对比原字段和新字段聚合结果，校验最大误差和报表展示精度"
    rollback_plan: "保留原字段或原表，回退视图指向原字段"
    execution_policy: manual_only
```

DDL 建议不能直接进入自动执行路径。即使建议由工具生成，也必须经过人工评审、业务口径确认、灰度验证和回滚方案确认。

参数渲染必须类型安全：

- 字符串参数需要做 SQL literal 转义。
- 数值参数需要校验格式，禁止把非数值内容渲染到数值位置。
- 日期参数需要校验格式和边界。
- `IN` 列表需要逐项校验和转义，不能把原始片段整段拼接。
- 缺失参数、重复冲突参数、非法参数都必须触发回退原 SQL。

模板引擎限制：

- 只支持白名单占位符替换。
- 不支持表达式执行。
- 不支持模板内动态 SQL 拼接。
- 不支持读取环境变量、文件或网络资源。
- 模板中的每个占位符必须在参数 schema 中定义。
- 抽取出来但模板未使用的参数必须记录警告；如果规则标记为 `strict_params=true`，则直接回退原 SQL。
- 未知占位符、缺失参数、类型不匹配、模板 checksum 不一致都必须失败并回退原 SQL。

已验证提速 SQL 模板示例：

```sql
select
  "业务类型",
  "货物中类",
  sum("成本") as "成本合计"
from optimized_report_table
where "年份" = '{{ 年份 }}'
  and "机械类型" in ({{ 机械类型 }})
group by
  "业务类型",
  "货物中类";
```

# 八、结果一致性校验

SQL 改写的最大风险是结果不一致。因此工具需要提供 `validate` 命令，对原 SQL 和改写 SQL 的结果做校验。

校验内容：

- 列数量一致。
- 列名一致。
- 列顺序一致。
- 字段类型兼容。
- 行数一致。
- 结果集合按 bag / multiset 语义一致。
- 数值误差在允许范围内。

校验方式：

1. 执行原 SQL，保存结果。
2. 执行改写 SQL，保存结果。
3. 对两个结果集做 bag / multiset 语义比较，重复行必须参与比较。
4. 输出是否一致、差异行数、最大数值误差。

对于不保证排序的 SQL，校验时不能直接按返回顺序比较，需要按所有列或指定 key 排序后比较。

`validate` 判定规则：

- 重复行：按 bag / multiset 语义比较，不能用简单 set 去重。
- `NULL`：两个 `NULL` 视为相等；`NULL` 与空字符串、0 不相等。
- `decimal` / `numeric`：按配置的绝对误差和相对误差阈值比较。
- 浮点数：按配置误差比较，并记录最大误差。
- `timestamp`：必须明确时区处理方式；未配置时区策略时不允许自动通过。
- 无序结果：使用全列排序或指定 key 排序后比较。
- 超大结果集：超过 `max_rows_in_memory` 时，使用流式 hash / 分片比较；第一版如果未实现流式比较，则返回 `validate_unsupported_large_result`。
- 查询超时：校验失败，规则不得进入自动改写候选。
- 执行错误：校验失败，记录错误摘要。
- 类型不兼容：校验失败。

`validate` 执行约束：

- 只允许校验只读 `SELECT`。
- 使用只读事务。
- 设置 `statement_timeout`。
- 禁止执行 DDL、DML、`COPY`、`CREATE TEMP TABLE` 等有副作用语句。
- 审计日志不得记录明文 DSN 密码。

校验通过后，规则才允许进入自动改写候选状态。校验失败的规则只能保留为分析材料，不能作为自动改写规则启用。

性能验证除了执行耗时，还必须为进入 `candidate` 的 SQL 保存 `EXPLAIN ANALYZE` 或等价执行计划证据。执行计划证据缺失时，SQL 最多只能标记为 `needs_review` 或 `dry_run_only`。

进入 `candidate` 的 SQL 应重点观察：

- 是否减少底层大表扫描次数。
- 是否减少扫描宽度和中间结果宽度。
- 是否减少 `Redistribute Motion` / `Broadcast Motion` 的数据量。
- 是否减少 `HashAgg`、`Sort` 的 spill 或 workfile。
- 是否避免隐式类型转换导致的表达式计算。
- 是否让关键过滤条件更早生效。

如果耗时下降但执行计划显示明显依赖偶然缓存、计划不稳定或并发下可能放大资源消耗，应标记为 `needs_review`，不直接进入候选规则库。

`validate` 与 `rewrite` 的关系：

- `rewrite` 可以在 `--dry-run` 模式下对未通过校验的规则生成候选 SQL，但输出状态必须是 `dry_run_only`。
- 没有 `validation.status=passed` 的规则，即使命中，也不能输出“建议进入自动改写候选规则库”。
- 只有 `validation.status=passed` 且 `enabled=true` 且人工确认的规则，才允许作为自动改写候选。

客户验收时，每条目标 SQL 都应形成一条校验证据：

- 原 SQL 文件。
- 改写后 SQL 文件。
- 命中的规则名。
- 抽取到的参数。
- 原 SQL 返回行数。
- 改写 SQL 返回行数。
- 结果一致性结论。
- 最大数值误差。
- 原 SQL 耗时。
- 改写 SQL 耗时。
- 是否建议进入自动改写候选规则库。

# 九、审计和回退

每次改写都应记录审计信息。

审计字段：

- 时间。
- 原 SQL 文件或 SQL 摘要。
- 规范化 SQL 指纹。
- 是否命中规则。
- 命中的规则名。
- 抽取到的参数。
- 是否执行改写。
- 回退原因。
- 输出 SQL 文件路径。
- 校验结果。
- 原 SQL 耗时。
- 改写 SQL 耗时。
- 是否建议进入自动改写候选规则库。
- 规则版本。
- 模板 checksum。
- 工具版本。

审计日志禁止记录：

- 明文 DSN 密码。
- 未脱敏的认证信息。
- 与客户安全策略冲突的敏感参数。

回退策略：

- 未命中规则：返回原 SQL。
- 指纹冲突：返回原 SQL。
- 参数抽取失败：返回原 SQL。
- 模板渲染失败：返回原 SQL。
- 校验失败：标记规则不可进入自动改写候选规则库。
- 运行期执行失败：后续代理模式下回退原 SQL。

第一版 CLI 阶段虽然不承载线上流量，但仍需要把回退逻辑实现清楚，因为后续 HTTP 服务或协议代理都会复用同一套规则和改写引擎。

## 9.1 回退输出契约

回退时必须满足以下输出契约：

| 场景 | stdout | `--out` 文件 | exit code | audit action | 原 SQL 是否字节级保持 |
|---|---|---|---:|---|---|
| 未命中规则 | 原 SQL | 写入原 SQL | 0 | `original` | 是 |
| 多规则冲突 | 原 SQL | 写入原 SQL | 0 | `original` | 是 |
| 参数抽取失败 | 原 SQL | 写入原 SQL | 0 | `original` | 是 |
| 模板渲染失败 | 原 SQL | 写入原 SQL | 0 | `original` | 是 |
| 规则未通过校验且非 dry-run | 原 SQL | 写入原 SQL | 0 | `original` | 是 |
| 输入文件不可读 | 错误信息 | 不写入 | 2 | `error` | 不适用 |
| 配置或规则文件非法 | 错误信息 | 不写入 | 2 | `error` | 不适用 |

对于业务安全回退场景，命令本身不应返回非 0，因为它成功完成了“保持原 SQL”的动作。只有工具自身无法运行、输入不可读、配置非法等工程错误才返回非 0。

# 十、CLI 命令设计

## 10.1 `fingerprint`

用于查看 SQL 规范化结果和指纹。

```bash
sqlopt fingerprint --sql-file samples/raw/bi_001.sql
```

输出内容：

- 规范化 SQL。
- SQL 指纹。
- 识别到的表。
- 识别到的过滤字段。

## 10.2 `rewrite`

用于生成已验证提速 SQL。

```bash
sqlopt rewrite --sql-file samples/raw/bi_001.sql --out samples/rewritten/bi_001.sql
```

输出内容：

- 是否命中规则。
- 命中规则名。
- 抽取参数。
- 是否改写。
- 输出文件路径。
- 回退原因。
- 是否建议进入自动改写候选规则库。

## 10.3 `explain`

用于解释为什么某条 SQL 命中或没有命中规则。

```bash
sqlopt explain --sql-file samples/raw/bi_001.sql
```

输出内容：

- 候选规则。
- 匹配成功项。
- 匹配失败项。
- 最终动作。

## 10.4 `validate`

用于连接数据库验证原 SQL 和改写 SQL 的结果一致性。

```bash
sqlopt validate --sql-file samples/raw/bi_001.sql --rewritten-file samples/rewritten/bi_001.sql --dsn postgres://user:pass@host:5432/db
```

## 10.5 `advise`

用于输出 DDL 和物理设计建议。该命令只生成建议，不执行 DDL。

```bash
sqlopt advise --sql-file samples/raw/bi_001.sql --schema-file samples/schema/customer_schema.sql --plan-file samples/plans/bi_001.plan
```

输出内容：

- 字段类型建议。
- 建表或预聚合表建议。
- 分布键、排序键、分区键建议。
- 索引和统计信息建议。
- 每条建议的收益、风险、验证方式和回滚方案。

# 十一、配置文件

全局配置示例：

```yaml
project: customer_case
rules_path: rules
templates_path: templates
audit_path: logs/audit.jsonl
rewrite:
  default_action: original
  allow_auto_rewrite: false
validate:
  numeric_tolerance: 0.000001
  max_rows_in_memory: 100000
  statement_timeout: 300s
  timezone: Asia/Shanghai
advisor:
  execution_policy: manual_only
  require_plan_for_candidate: true
```

`allow_auto_rewrite` 默认必须为 `false`。只有规则通过 `validate`、人工确认、并进入下一阶段接入验证后，才允许按项目或规则显式打开。

# 十二、测试策略

第一版至少需要以下测试：

- SQL 规范化单元测试。
- SQL 指纹稳定性测试。
- 规则加载测试。
- 规则命中和未命中测试。
- 参数抽取测试。
- 模板渲染测试。
- 回退逻辑测试。
- CLI 输出测试。
- 结果校验逻辑测试。
- SQL literal 转义测试。
- 参数非法输入回退测试。
- CLI golden 输出测试。
- DDL / 物理设计建议输出测试。

测试矩阵：

| 模块 | 必测输入 | 预期结果 |
|---|---|---|
| `normalize` | 空格、大小写、注释、字符串常量、数字常量、中文引号字段 | 同类 SQL 规范化稳定，不同结构不合并。 |
| `fingerprint` | 同模板不同常量 SQL | 指纹一致。 |
| `fingerprint` | 近似但 `group by` 或聚合不同 SQL | 指纹或结构校验不命中。 |
| `rules` | 禁用规则 | 不命中。 |
| `rules` | 多规则同指纹 | 规则加载失败或运行时回退，不能任选一条。 |
| `extract` | 合法字符串、日期、数值、`IN` 列表 | 参数按 schema 输出。 |
| `extract` | 非法年份、超长字符串、列表超限、重复冲突参数 | 回退原 SQL。 |
| `rewrite` | 缺失参数、未知占位符、模板 checksum 不一致 | 回退原 SQL。 |
| `rewrite` | SQL literal 中含单引号 | 正确转义。 |
| `rewrite` | `IN` 列表包含注入式片段 | 拒绝并回退原 SQL。 |
| `validate` | 重复行 | 按 bag / multiset 语义比较。 |
| `validate` | `NULL`、空字符串、0 混合 | 正确区分。 |
| `validate` | 小数误差在阈值内 / 阈值外 | 分别通过 / 失败。 |
| `validate` | 无序结果 | 排序或 key 比较后判断一致。 |
| `validate` | 类型不兼容 | 失败。 |
| `advisor` | SQL + DDL + 执行计划 | 输出字段类型、建表、索引或统计信息建议，且每条建议包含风险和验证方式。 |
| `advisor` | 缺少执行计划 | 允许输出一般建议，但不得把建议作为 `candidate` 证据。 |
| CLI | `rewrite` 命中、未命中、回退 | stdout、out 文件、exit code、audit 符合契约。 |

某客户场景专项测试：

- 准备 13 条 BI 原始 SQL 样本。
- 为每类 SQL 建立规则。
- 为每类 SQL 准备已验证提速模板。
- 对每条 SQL 做 `fingerprint`、`rewrite`、`explain`。
- 对已准备好数据库环境的 SQL 做 `validate`。
- 每条成功改写 SQL 都要保留原 SQL、改写 SQL、规则文件和校验结果。

# 十三、后续演进

第一阶段：CLI + 改写引擎。

- 目标是把某客户场景 SQL 模板识别和改写跑通。
- 重点验证规则库、参数抽取、模板渲染和结果一致性。

第二阶段：HTTP 服务。

- 提供接口接收原 SQL，返回改写 SQL。
- 方便和外部平台或网关集成。

第三阶段：PostgreSQL 协议代理。

- BI 连接代理，代理连接 YMatrix。
- 代理实时识别 SQL，命中后自动改写并执行。
- 需要处理认证、SSL、prepared statement、事务、游标、错误回退等协议细节。
- 只有第一阶段和第二阶段验证稳定后，才进入协议代理阶段。

第四阶段：规则管理平台。

- 提供规则启停、灰度、审计查询、命中统计。
- 支持按项目、用户、数据库、报表 ID 控制改写范围。

# 十四、风险控制

主要风险：

- 误命中导致 SQL 被错误改写。
- 参数抽取不完整导致结果口径变化。
- 已验证提速 SQL 模板和原 SQL 结果不一致。
- BI 平台生成 SQL 轻微变化导致规则失效。
- 后续代理模式下需要处理 PostgreSQL 协议复杂性。
- 只用正则理解完整 SQL，导致误命中或漏掉语义差异。
- 审计日志泄露连接串或敏感参数。

控制措施：

- 第一版规则匹配从严格开始，宁可不命中也不误改。
- 所有自动改写规则必须先通过结果一致性校验。
- 默认保留原 SQL 回退路径。
- 审计日志必须记录每次改写细节。
- 规则启用前需要人工确认。
- 对外承诺只覆盖已确认的 SQL 模板，不承诺未知 SQL 自动优化。
- 新 SQL 模式必须经过采样、分析、规则开发、结果校验、灰度启用。
- 禁止只靠正则做完整 SQL 结构判断。
- `validate` 必须使用只读事务和超时控制。
- 性能对比必须固定数据库版本、数据快照、执行账号、并发条件、缓存策略和重复执行次数。

# 十五、第一版交付物

第一版完成后应包含：

- Go 项目骨架。
- `sqlopt` CLI。
- SQL 规范化模块。
- SQL 指纹模块。
- 规则加载和匹配模块。
- 参数抽取模块。
- 模板渲染模块。
- 基础审计日志。
- 某客户场景规则样例。
- 某客户场景已验证提速 SQL 模板样例。
- 单元测试。
- README 使用说明。
- 客户验收报告样例。
- 改写审计日志样例。
- DDL / 物理设计建议报告样例。

研发验收标准：

- 能读取 SQL 文件。
- 能生成稳定指纹。
- 能加载规则文件。
- 能识别命中和未命中。
- 能抽取规则定义的参数。
- 能生成改写后的 SQL。
- 未命中或失败时能返回原 SQL。
- 能输出清晰的命中解释和审计信息。
- 参数渲染具备基础类型安全和转义能力。
- 单元测试覆盖核心改写路径和失败回退路径。
- `advise` 能输出只读、手工执行的 DDL / 物理设计建议，并包含风险、验证方式和回滚方案。

客户验收标准：

- 验收范围锁定为某客户场景 13 条已采集 BI SQL 样本，或双方书面确认的子集。
- 每条纳入验收的 SQL 都有 `SQL ID`、报表名、业务方、原 SQL 文件、规则名、模板名和参数记录。
- 覆盖率按验收清单计算，不按未知 SQL 计算。
- 每条目标 SQL 都通过结果一致性校验，或明确标记为 `dry_run_only` / `不可进入自动改写候选规则库`。
- 性能对比必须在固定测试环境中执行，至少记录数据库版本、数据快照、执行账号、并发条件、冷热缓存策略、重复执行次数、超时设置、平均耗时和 P95 耗时。
- 通过标准分为 `passed`、`failed`、`dry_run_only`、`not_in_scope` 四类。
- 未命中 SQL 或异常 SQL 能保持原样，不改变业务结果。
- 规则可关闭，改写行为可追溯。
- 输出一份可交付的性能对比和下一阶段接入建议报告。

默认验收阈值：

| 验收项 | 默认通过阈值 |
|---|---|
| 样本登记覆盖 | 13 条已采集 SQL 必须 13/13 进入验收清单。 |
| `fingerprint` 覆盖 | 13 条已采集 SQL 必须 13/13 能输出指纹和结构摘要。 |
| `explain` 覆盖 | 13 条已采集 SQL 必须 13/13 能输出命中、未命中或不纳入范围的明确原因。 |
| 改写候选覆盖 | 第一版至少 3 条重点慢 SQL 或至少 1 类慢 SQL 模板达到 `candidate`；如果双方书面确认子集，则按子集阈值验收。 |
| 回退覆盖 | 未命中、规则冲突、参数抽取失败、模板渲染失败场景必须 100% 返回原 SQL。 |
| 结果一致性 | 进入 `candidate` 的 SQL 必须 100% 通过 `validate`。 |
| 审计覆盖 | 每次 `rewrite` / `validate` 必须 100% 产生日志记录。 |
| 执行计划证据 | 进入 `candidate` 的 SQL 必须 100% 保存 `plan_evidence`。 |
| 数据库友好检查 | 进入 `candidate` 的 SQL 必须 100% 有结构化 `database_friendly_checks`，且至少一项为 `yes`。 |

性能达标线：

判定按以下优先级执行，命中高优先级后不再继续向下判断：

| 优先级 | 判定 | 条件 |
|---:|---|---|
| 1 | `not_in_scope` | 不属于第一版验收范围，或 SQL 结构与已采集模板不一致。 |
| 2 | `dry_run_only` | 缺少数据库校验环境、结果一致性证据或性能证据。 |
| 3 | `reject` | 结果不一致、执行失败、业务方不认可结果口径，或平均耗时无改善。 |
| 4 | `candidate` | 满足结果一致、性能、计划证据和数据库友好检查的全部候选条件。 |
| 5 | `needs_review` | 结果一致且有一定分析价值，但未达到 `candidate`，也未触发 `reject`。 |

| 判定 | 标准 |
|---|---|
| `candidate` | 结果一致；平均耗时下降不低于 20%；P95 耗时不劣于原 SQL；重复执行不少于 3 次；`plan_evidence` 非空；`database_friendly_checks` 至少一项为 `yes`；无明显计划不稳定或并发放大风险。 |
| `needs_review` | 结果一致，平均耗时有改善但低于 20%，或缺少执行计划证据，或数据库友好检查全为 `no` / `not_applicable`，或执行计划存在稳定性/并发风险。 |
| `reject` | 结果不一致、执行失败、业务方不认可结果口径，或结果一致但平均耗时无改善。 |
| `dry_run_only` | 能识别和生成候选 SQL，但缺少数据库校验环境、结果一致性证据或性能证据。 |
| `not_in_scope` | 不属于第一版验收范围，或 SQL 结构与已采集模板不一致。 |

如果客户和项目组另有书面约定，以书面约定覆盖默认阈值；没有书面约定时，按上述默认阈值验收。

常见问题：

| 问题 | 回答 |
|---|---|
| 新报表 SQL 结构变了怎么办？ | 重新采样、生成指纹、评估规则，不能假定旧规则自动适配。 |
| 未命中的 SQL 是否影响原查询？ | 不影响，第一版返回原 SQL。 |
| 规则关闭后是否立即回退？ | 是，禁用规则后对应 SQL 不再改写。 |
| 第一版 CLI 如何用于客户演示？ | 用样本 SQL 演示 `fingerprint`、`explain`、`rewrite`、`validate`，并输出验收报告。 |
| 结果一致性由谁确认？ | 工具提供校验结果，业务方确认口径，研发方确认技术实现。 |
| 性能提升不达标但结果一致怎么办？ | 平均耗时有改善但低于 20% 时标记为 `needs_review`；平均耗时无改善时标记为 `reject`。 |
| 没有数据库验证环境能否验收？ | 只能验收识别和改写能力，不能验收结果一致性和性能收益。 |

# 十六、双 worker 监督机制

本项目开发过程中保留两个固定监督视角：

1. 客户需求 worker。
2. 研发质量 worker。

客户需求 worker 负责审查：

- 当前功能是否能支撑客户价值和交付验收。
- 文档表达是否避免过度技术化。
- 是否清楚说明覆盖范围、性能收益、结果一致性和回退方案。
- 是否存在对客户过度承诺，例如“任意 SQL 自动优化”或“自动生成最优 SQL”。

研发质量 worker 负责审查：

- 需求边界是否被严格收敛。
- 规则匹配、参数抽取、模板渲染和回退逻辑是否可靠。
- 是否存在误命中、SQL 注入式模板拼接、结果口径变化等风险。
- 测试是否覆盖成功改写、未命中、抽取失败、渲染失败和校验失败。

监督节奏：

- 设计文档更新后，两类 worker 各审查一次。
- 实现计划生成后，两类 worker 各审查一次。
- 每个核心模块实现后，至少让研发质量 worker 审查一次。
- 每次准备对外演示或交付材料前，至少让客户需求 worker 审查一次。
- 两个 worker 出现分歧时，以“结果正确、风险可控、客户可验收”为优先级排序。
